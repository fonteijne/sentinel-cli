## Management Summary — Non-technical

Today’s work focused on tightening how the hosted Sentinel endpoint is protected when it’s exposed through Cloudflare. The main goal is to make it safer and easier for multiple people (and automated jobs) to use the service without relying on fragile “only allow certain IP addresses” rules.

The deployment guide was expanded to recommend Cloudflare Access as the standard “front door” for teams. This gives a clear, auditable login layer for each operator (via email or single sign-on), while Sentinel itself keeps a second layer of protection using its existing bearer token.

There is also an outlined follow-up option (Authenticated Origin Pulls) to prevent someone from bypassing Cloudflare and reaching the origin directly.

## Management Summary — Technical

All changes landed on `experimental/command-center-06-production-exposure` and revolve around the production exposure pattern for the self-hosted deployment behind Cloudflare. The docs extend `docs/deploy.md` with a substantially richer Cloudflare Access section, reframing it as the recommended multi-operator ingress layer. This now explicitly models a two-gate architecture (Cloudflare Access at the edge for per-user identity, plus the plan-05 in-service bearer token behind Traefik), and adds concrete operator lifecycle procedures (onboarding via policy changes; offboarding with session revocation to avoid the 24-hour cookie window).

The Access guidance is made more operationally precise: service tokens are treated as first-class credentials for automation with an explicit “one token per actor” recommendation, and a rotation table enumerates which knobs affect which clients. The documentation also acknowledges the current limitation that Sentinel’s plan-05 bearer token is still a shared secret, so per-user attribution is only available in Cloudflare Access logs until a future plan introduces per-user tokens.

On the Compose side, today’s commit sequence explores and then simplifies several Traefik-level mitigations for Cloudflare “orange cloud” deployments. The branch initially adds forwarded-header trust for Cloudflare edge CIDRs and an ipallowlist exclusion list so the operator allowlist can evaluate the true client IP from X-Forwarded-For, then attempts to block origin bypass with a second Cloudflare-only ipallowlist middleware. This is then reverted/removed for the Docker Desktop Mac topology because vpnkit source-NAT collapses TCP source addresses to the bridge gateway, making any TCP-source-based Cloudflare-only check ineffective and undermining trustedips-by-CF-range. The final state removes Traefik IP allowlisting and forwarded-headers trust entirely, relying on Cloudflare Access for identity gating and leaving Traefik primarily as TLS termination and reverse proxy to `sentinel-serve`.
