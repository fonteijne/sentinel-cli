## Management Summary — Technical (not familiar with sentinel-cli)

### What is sentinel-cli?

Sentinel-CLI is a Python tool that orchestrates Claude AI agents (via the Claude Agent SDK) to autonomously work Jira tickets end-to-end: **plan → execute → security review → debrief**. Each ticket runs in an isolated Git worktree; results are pushed as GitLab merge requests. It targets a solo developer on a MacBook Pro running against a corporate Jira Server and self-hosted GitLab, with 3–8 parallel tickets in flight.

The current stack: Python + Click CLI, Claude Agent SDK, `poetry`, Docker/Lando for Drupal environments, Qdrant-adjacent session tracking, a SQLite-backed beads state store.

**Commit `046388f8`** — *"docs: add PRD for Sentinel service + command center"* — 15:08 UTC

**File added:** `docs/PRD-dashboard-service.md` (+498 lines)

A comprehensive PRD proposing the evolution of sentinel-cli into a **long-running FastAPI service with a React command-center UI**. Pure documentation — no source code touched.

**Proposed architecture:**

| Layer | Technology |
|---|---|
| Backend API | FastAPI + Uvicorn (`src/api/` — new module) |
| Frontend | React 18 + Vite + TypeScript + shadcn/ui + TanStack Query/Router |
| Real-time | WebSocket at `/ws/board`, `/ws/runs/{id}`, `/ws/inbox` via in-process asyncio pub/sub |
| Persistence | Existing `~/.sentinel/` state + new SQLite `service.db` (`runs` + `review_inbox` tables) |
| Packaging | `sentinel-service` console script + optional `docker-compose.service.yml`, served on `localhost:3939` |

**Four MVP modules:**

| Module | Purpose |
|---|---|
| A — Ticket Board | Kanban of Sentinel lifecycle states (not Jira statuses). One-click plan/execute/revise/debrief/reset per card. Feeds from `jira_client` + `beads_manager`. |
| B — Worktree Manager | Live list of all worktrees with `git status --porcelain=v2`, VS Code/Cursor deep-link open, reset, archive. Feeds `worktree_manager`. |
| C — Live Run Viewer | Three-pane viewer: phase timeline / tool-call stream / attachments + MR diff. WebSocket stream from `session_tracker` via inotify/watchdog. Abort/pause/note controls. |
| D — Review Inbox | Unified GitLab MR comment + Jira comment feed, polled every 60 s. One-click "Revise with this feedback" → triggers `execute --revise`. |

**Key design decisions:**
- All existing managers (`worktree_manager`, `jira_client`, `gitlab_client`, `session_tracker`, `beads_manager`) reused as library — zero rewrite.
- Loopback-only (`127.0.0.1`), no auth at MVP; CORS locked to `localhost:5173` / `127.0.0.1:3939`.
- Runs as in-process `asyncio` tasks — no Celery/Redis. Supports ~10 concurrent runs.
- CLI remains fully functional and unchanged.

**Milestones:** M0 (skeleton) → M1 (Worktree UI) → M2 (Runs persistence) → M3 (Live tailing) → M4 (Ticket Board) → M5 (Review Inbox) → M6 (Hardening).
